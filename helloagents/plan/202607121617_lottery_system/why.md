# 变更提案: 双奖池邀请抽奖系统

## 需求背景

站点已有邀请码归因、充值履约、激活码兑换、余额和订阅能力，但缺少将这些能力组合成用户增长活动的抽奖入口。新增抽奖系统后，管理员可以配置普通与豪华两个奖池、奖品概率和邀请奖励规则，用户可通过日常机会及邀请新用户获得额外抽奖次数。

## 产品分析

### 目标用户与场景
- **普通用户:** 使用周期赠送次数抽奖，查看奖品概率和历史结果。
- **邀请用户:** 分享现有注册链接，在被邀请人注册、兑换激活码或充值达标后获得额外次数。
- **管理员:** 配置两个奖池、奖品、规则和活动时间，并查询次数及中奖审计记录。

### 价值主张与成功指标
- **价值主张:** 复用现有邀请和权益体系，以可审计、可配置的抽奖活动提高注册、激活和充值转化。
- **成功指标:** 抽奖结果无重复扣次或超发库存；规则事件幂等；余额和订阅奖品自动到账；管理员可追溯每次次数变化及抽奖结果。

### 人文关怀
公开每个奖品的准确概率，不隐藏未中奖概率；动画支持减少动态效果偏好；不采集设备指纹或额外个人信息。涉及充值的规则仅使用现有订单和退款数据。

## 变更内容

1. 增加固定的普通、豪华两个奖池，分别配置启停、起止时间、每日或每周周期及每周期基础次数。
2. 增加余额、订阅两类奖品，支持概率、固定或无限库存、启停、排序和可选卡面图片。
3. 复用现有邀请关系，增加注册、首次激活码兑换、单笔/累计充值三类奖励规则。
4. 增加用户次数账户、不可变次数流水和抽奖记录，区分周期基础次数与长期有效的额外次数。
5. 增加用户抽奖页和独立管理页；用户端采用横向卡牌轮带动画，管理端维护配置及审计记录。

## 影响范围

- **模块:** 邀请注册、兑换码、支付履约与退款、余额、订阅、用户导航、管理端。
- **后端:** 新增 lottery domain/service/repository/handler；调整 affiliate、redeem、payment 的成功事件接入点及 Wire 注入。
- **前端:** 新增用户抽奖页、抽奖 API、轮带组件、管理页、路由、侧边栏和中英文文案。
- **数据:** 新增奖池、奖品、规则、次数账户、次数流水和抽奖记录表；不修改现有邀请归属数据。
- **API:** 新增用户抽奖摘要/执行/历史接口，以及管理端奖池、奖品、规则和记录接口。

## 核心规则

### 需求: 双奖池和周期次数
**模块:** 抽奖核心

普通和豪华是两条固定、独立的奖池配置，管理员不能创建第三种奖池，但可分别修改周期及次数。

#### 场景: 周期刷新
- 每个奖池可选择每日或每周周期，并设置每周期基础次数。
- 以站点时区零点生成周期键；周周期默认从周一开始。
- 未使用的基础次数不累计，用户访问或抽奖时惰性刷新，无需定时任务。
- 邀请规则发放的额外次数长期累计，抽奖时优先消耗即将过期的基础次数。

#### 场景: 活动可用性
- 每个奖池支持启用开关和可选开始、结束时间。
- 未开始、已结束或停用时不可抽奖，但历史记录仍可查询。

### 需求: 奖品和概率
**模块:** 抽奖核心

#### 场景: 配置奖品
- 奖品仅支持余额和订阅；订阅奖品配置现有订阅分组和有效天数。
- 每个奖品可选择无限库存或固定库存，并可配置名称、说明、排序及可选卡面图片。
- 管理端上传图片复用现有 `ImageUpload` Data URL 方案；后端限制安全图片 MIME 和大小，未上传时显示类型默认图标。

#### 场景: 计算结果
- 每个奖池启用奖品的概率总和不得超过 100%，不足部分为未中奖。
- 用户端展示奖品准确概率，不展示剩余库存。
- 库存耗尽、停用的奖品不再中奖，其原概率归入未中奖，不按比例分配给其他奖品。
- 服务端使用安全随机数确定结果，前端动画不参与概率计算。

### 需求: 邀请奖励规则
**模块:** 抽奖规则

#### 场景: 规则目标
- 每条规则至少选择普通或豪华一个奖池，也可同时选择两个奖池，并分别设置增加次数。
- 同一种事件可配置多条规则；所有满足条件的规则独立生效并叠加。
- 注册规则可分别奖励邀请人或被邀请人；激活码和充值规则奖励邀请人。
- 禁止自邀，并通过来源事件、规则、受益人和档位唯一键保证重复通知不重复发放。

#### 场景: 注册和激活码
- 直接复用现有邀请码、注册链接及永久邀请归属。
- 被邀请人绑定邀请关系成功后计算注册规则。
- 被邀请人首次成功兑换任意非邀请类型激活码时计算激活码规则；后续兑换不再触发此类规则。

#### 场景: 充值门槛
- 每条充值规则可选择单笔或累计站内充值金额，并配置门槛和“仅首次有效”。
- 单笔模式按每个完成订单判断；允许重复时，每个达标订单分别奖励。
- 累计模式按扣除已退款金额后的累计充值计算；允许重复时，每跨过一个门槛倍数奖励一次。
- 多条充值规则可同时命中并叠加。

#### 场景: 充值退款
- 订单最终退款后，对该订单导致的有效档位重新计算并写入冲正流水。
- 仅扣回用户尚未使用的额外次数；不足部分记录审计，不追回已抽中奖品。
- 冲正具备幂等键，重复退款通知不会重复扣减。

### 需求: 自动发奖
**模块:** 余额与订阅

#### 场景: 余额奖品
- 在抽奖事务内直接调用现有用户余额调整入口，并记录奖品快照。

#### 场景: 订阅奖品
- 系统在抽奖事务内生成一次性订阅激活码，并立即以中奖用户身份兑换。
- 兑换统一经过现有激活码权益入口；同组有效订阅按现有逻辑叠加天数，不另建订阅发放逻辑。
- 抽奖生成的内部激活码不触发“首次兑换激活码”邀请规则，避免递归奖励。
- 任一步骤失败时回滚次数、库存、中奖记录、激活码和权益，用户可重试。

### 需求: 用户与管理界面
**模块:** 前端

#### 场景: 用户抽奖
- 普通和豪华使用页签切换，展示周期基础次数、长期额外次数、奖品列表和准确概率。
- 抽奖时背景虚化，前景卡牌横向循环、减速并停在服务端返回结果上。
- 请求期间禁止重复操作；减少动态效果模式直接展示结果。
- 用户可查看自己的抽奖历史，并跳转现有邀请页复制分享链接。

#### 场景: 管理配置
- 独立抽奖管理页维护奖池、奖品、规则，不继续扩展现有大型设置页。
- 管理员可按用户、奖池、结果和时间查询抽奖记录与次数流水。
- 历史记录保存奖品及规则快照，后续修改配置不改变既有记录。

## 范围外

- 不增加第三种奖池、十连抽、保底、积分兑换、实物奖品或人工发奖。
- 不实现通用营销规则引擎、消息总线、设备指纹或同 IP 限制。
- 不自动追回退款前已经抽中的余额或订阅。
- 不新增前端动画、随机数或上传依赖。

## 验收标准

1. 并发抽奖不会把同一用户次数扣成负数，也不会超出固定库存。
2. 相同注册、兑换、支付和退款事件重复处理时，次数流水只生效一次。
3. 概率总和不足 100% 时可以未中奖，库存耗尽不会提高其他奖品概率。
4. 订阅奖品产生已使用的一次性激活码记录，并与抽奖记录关联。
5. 抽奖事务故障时不留下已扣次数、已减库存或部分到账状态。
6. 用户端动画结束位置与服务端结果一致，移动端和桌面端均无布局重叠。

## 风险评估

- **并发超发:** 使用用户次数行锁、奖品条件更新和数据库事务；以并发集成测试验证。
- **规则重复发放:** 使用来源事件、规则、受益人、奖池及档位组成唯一约束。
- **订阅部分提交:** 重构兑换码内部事务入口，使其复用抽奖外层事务，并将缓存失效延后到提交后。
- **退款已消费:** 只冲回可用额外次数，差额写入审计，不修改历史奖品。
- **图片输入:** 服务端校验 Data URL MIME、编码和体积，不允许 SVG 或外部 URL。
