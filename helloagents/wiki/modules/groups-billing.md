# 分组与计费

## 目的
以分组隔离平台、账号池、用户入口和计费规则。

## 规范

### 需求: 媒体价格
**模块:** 分组与计费

#### 场景: 视频计费
- 视频价格绑定分组和分辨率。
- 分组倍率或视频独立倍率作用于基础视频价格。
- `per_second` 按价格乘规范化时长，`per_request` 每次只计一个价格单位。
- `/v1/videos` 与 `/v1/video/create` 未传时长时默认 4 秒、未传分辨率时默认 720p。
- 金额写入必须使用请求级幂等键。

### 需求: 余额扣费
**模块:** 分组与计费

#### 场景: 异步任务失败
- 本次视频平台只处理标准余额，不处理订阅额度退款。
- 扣余额和任务快照必须在同一事务提交，任一写入失败则整体回滚。
- 明确失败终态的退款只能执行一次，并同步失效余额缓存；网络错误和未知状态继续重试。

### 需求: 独立订阅权益
**模块:** 分组与计费

#### 场景: 重复获得同一分组订阅
- 每次购买、兑换、抽奖或管理员发放都创建独立订阅记录。
- 每张订阅从获得时立即开始计算自己的有效期，既有订阅的时间和用量不变。
- 注册和身份绑定的默认权益仍通过幂等入口发放，重试不得重复创建。

#### 场景: 请求选择订阅
- 查询当前用户和分组的所有有效订阅，按 `expires_at ASC, id ASC` 排序。
- 跳过当前日、周或月额度已满的记录，使用第一张仍有额度的订阅。
- 多日订阅的窗口重置后可重新参与选择；1 日卡的日额度为一次性额度。
- 单次请求固定记入开始时选中的 `subscription_id`，不跨两张订阅拆账；后续请求再切换。

#### 场景: 订阅订单退款
- 新支付订单保存其发放的 `subscription_id`，退款只调整该订阅。
- 历史未关联订单不得推测性扣减其他权益，需要管理员强制确认。

### 需求: API Key 有序多分组容灾
**模块:** API Key、分组与计费

#### 场景: 用户配置候选顺序
- 每个 Key 必须配置 1-5 个去重、同平台且当前用户有权使用的分组。
- 用户拖拽顺序是严格消费与容灾顺序，余额组和订阅组可自由混排。
- 创建或编辑 Key 时，待添加分组的下拉选项显示分组默认倍率、用户专属倍率和峰值倍率信息。
- `group_id` 始终镜像第一组；旧客户端只提交该字段时替换为单分组链。

#### 场景: 请求选择与切换
- 每个新请求从第一组开始，不保存上次请求的切换位置，首组恢复后自动回切。
- 分组停用、余额不足、订阅耗尽/过期或无有效订阅时跳过；订阅组内部按最早到期且有额度的权益消费。
- 当前组无可调度账号，或连接错误、超时、429、5xx 在组内重试耗尽后，才推进下一组并重新检查计费资格。
- 参数错误、权限错误、客户端取消、响应已提交或异步任务可能已创建时不切换。
- 用量和扣费始终归属实际成功的 `group_id`、`subscription_id` 与账号。
