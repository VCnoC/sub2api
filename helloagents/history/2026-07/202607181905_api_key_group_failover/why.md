# 变更提案: API Key 有序多分组容灾

## 需求背景
当前每个 API Key 只绑定一个 `group_id`。分组内账号全部不可调度、上游持续限流/故障，或订阅额度耗尽时，请求会直接失败，无法使用用户拥有的其他同平台分组继续工作。

## 产品分析

### 目标用户与场景
- **用户群体:** 需要长时间无人值守调用 API 的个人和服务。
- **使用场景:** 用户为一个 API Key 配置 `余额1 → 余额2 → 尝鲜 → 月卡 → 余额3`，系统按顺序尝试。
- **核心痛点:** 单个分组夜间故障或额度耗尽会中断整个工作流。

### 价值主张与成功指标
- **价值主张:** 在不更换客户端 Key 的情况下，自动跨分组容灾。
- **成功指标:** 可配置 1-5 个同平台分组；可重试故障和计费资格耗尽时按序切换；实际成功分组正确计费和记账。

### 人文关怀
默认保持旧 Key 单分组行为，避免升级后产生意外消费；界面明确展示优先级和实际计费分组。

## 变更内容
1. 新增 API Key 与分组的有序关联表，并将现有 `group_id` 回填为第一优先级。
2. API Key 创建、更新、响应和认证缓存支持最多 5 个同平台候选分组。
3. 扩展现有账号 failover：当前分组耗尽后，在安全边界内切换下一分组。
4. 用户端支持多选、拖拽排序，并展示候选链。

## 影响范围
- **模块:** API Key、鉴权缓存、分组授权、订阅与余额资格、网关调度、用量计费、用户端 Key 管理。
- **文件:** Ent schema、migration、API Key repository/service/handler/cache、gateway failover、KeysView 及 API 类型与测试。
- **API:** 创建/更新新增有序 `group_ids`；响应同时保留兼容字段 `group_id`。
- **数据:** 新增 `api_key_groups`，回填现有非空 `api_keys.group_id`。

## 核心场景

### 需求: 用户配置有序候选分组
**模块:** API Key 管理
用户为一个 Key 选择 1-5 个同平台且自己有权使用的分组，并拖拽确定顺序。

#### 场景: 旧 Key 升级
- 旧 `group_id` 自动成为 `group_ids[0]`。
- 旧 API 客户端只提交 `group_id` 时仍创建或更新为单分组链。

### 需求: 请求按优先级跨组容灾
**模块:** 鉴权、调度与计费
每个新请求从第一分组重新开始，在当前分组不可用时依次尝试后续分组。

#### 场景: 分组计费资格不可用
- 余额不足、订阅耗尽/过期、分组停用或无有效订阅时跳过当前分组。
- 订阅分组内部继续使用最早到期且有额度的具体订阅。

#### 场景: 上游故障
- 当前分组账号重试耗尽后，连接错误、超时、429 和 5xx 可切换下一分组。
- 参数错误、权限错误、客户端取消不切换。
- 已向客户端输出内容或上游可能已创建异步任务时不切换。

#### 场景: 实际分组记账
- 成功请求按实际分组执行余额或订阅计费。
- 用量日志保存实际 `group_id`、`subscription_id` 和账号信息。

## 风险评估
- **风险:** 跨组重试增加时延或重复执行非幂等请求。
- **缓解:** 最多 5 组；只在明确可重试且响应未提交时切换；异步任务发送后禁止跨组。
- **风险:** 旧 `group_id` 与新关联表不一致。
- **缓解:** 关联表作为真实来源，服务层事务写入并同步首组兼容字段，增加迁移与一致性测试。
