# 技术设计: 合并官方 v0.1.160

## 技术方案

### 核心技术
- Git 三方合并与独立快照分支。
- Ent/Wire 官方生成流程。
- Go、Vitest、TypeScript 和 Vite 分层验证。

### 实现要点
- 从当前 `merge-upstream-v0.1.153` 创建 `merge-upstream-v0.1.160`，先提交完整脏工作区作为可恢复快照。
- GitHub Git 协议不可用时，从官方源码归档构造以 `v0.1.153` 为父的 `v0.1.160` 上游快照，再执行三方 merge 并校验完整文件差异。
- 普通源码按语义合并；Ent/Wire 生成文件不手工拼接，先解决源 schema/provider 后重新生成。
- 官方新增迁移保持 174-182；本地多订阅迁移改为 183/184。
- 对支付退款、API Key 鉴权、计费、订阅卡片和账号列表等重叠文件执行定向回归。

## 架构决策 ADR

### ADR-20260718-UPSTREAM-160-001: 使用快照分支执行三方合并
**上下文:** 当前工作区包含大量尚未提交的本地功能，直接合并会被 Git 阻止且难以恢复。
**决策:** 新建升级分支并提交完整快照；以官方源码归档构造带 `v0.1.153` 父提交的上游快照，再执行三方合并。
**理由:** 提供明确回滚点，并在 Git 网络不可用时保持官方差异与本地改动的共同祖先关系。
**替代方案:** 先 stash 后合并再恢复 → 拒绝原因: 大量未跟踪文件和跨模块改动使恢复冲突更难审计。
**影响:** 合并分支增加一个快照提交，但不会污染原 v0.1.153 分支。

### ADR-20260718-UPSTREAM-160-002: 生成代码统一重建
**上下文:** Ent 和 Wire 生成文件存在多处冲突，人工合并容易产生运行时不一致。
**决策:** 合并 schema、配置和 provider 源文件后运行项目现有生成命令。
**理由:** 生成器是这些文件的唯一可靠来源。
**替代方案:** 手工解决每个生成文件冲突 → 拒绝原因: 工作量大且容易遗漏字段或依赖。
**影响:** 生成结果可能产生较大机械 diff，需要审计源定义与输出一致。

## 安全与性能
- **安全:** 保留官方操作审计、会话绑定和 step-up 2FA；不连接生产数据库或服务。
- **数据:** 迁移只调整文件编号，不执行数据库迁移。
- **性能:** 保留本地账号用量批量加载和多订阅候选逻辑，同时接入官方调度与计费优化。

## 测试与部署
- **后端:** 生成检查、关键包测试、完整构建，条件允许时运行默认测试集。
- **前端:** 相关 Vitest、`pnpm typecheck`、Vite 生产构建。
- **部署:** 本次仅完成代码合并；生产迁移和发布另行执行。
