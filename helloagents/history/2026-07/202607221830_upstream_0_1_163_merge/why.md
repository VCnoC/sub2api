# 变更提案: 合并官方 v0.1.163

## 需求背景
当前定制分支以官方 v0.1.160 等价快照为基线，已包含有序 API Key 分组、生图错误透传、次数订阅和团队治理等本地能力。官方 v0.1.161-v0.1.163 累计 245 个提交，包含安全配置、Grok/OpenAI 网关、调度计费、移动端和依赖漏洞修复，需在不丢失本地功能的前提下升级。

## 变更内容
1. 引入官方 v0.1.161-v0.1.163 全部代码变更。
2. 保留本地视频、抽奖、工单、有序分组、次数订阅、团队治理和生图错误透传。
3. 将官方新增 migration 183-185 顺延为 188-190，不更改已在生产使用的本地 183-187。
4. 重新生成 Ent 与 Wire，并执行后端、前端和迁移回归验证。

## 影响范围
- **模块:** 网关、调度、计费、分组、安全设置、账号、订阅、管理端与用户端。
- **文件:** 官方 v0.1.160-v0.1.163 差异文件及 106 个本地重叠文件。
- **API:** 新增分组推理策略、客户端 IP 配置、异步生图存储和 Grok compact 能力。
- **数据:** 新增入口拒绝聚合、鉴权缓存失效 outbox 和分组推理策略字段。

## 核心场景

### 需求: 保留定制能力升级官方版本
**模块:** 全栈上游合并

#### 场景: 冲突文件同时包含官方与本地改动
- 以本地 v0.1.160 等价快照为共同父节点执行三方合并。
- 以业务语义同时保留两侧功能，不整文件选择 ours/theirs。
- 生成文件以 schema/provider 为真实源重建。

#### 场景: migration 编号与生产历史冲突
- 本地 183-187 文件名和校验和保持不变。
- 官方 183-185 顺延为 188-190，并更新测试和文档引用。

## 风险评估
- **风险:** 网关、计费与订阅重叠可导致重复扣费或容灾回归。
- **缓解:** 定向保留本地计费边界，运行请求次数、订阅、团队资金和分组容灾测试。
- **风险:** 迁移编号覆盖已执行文件会触发 checksum 失败或数据结构错乱。
- **缓解:** 只顺延未在本地生产执行的官方新迁移，并用隔离 PostgreSQL 验证全量迁移。
