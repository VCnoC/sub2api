# 变更提案: 恢复海外D未入库功能

## 需求背景
海外D散户站曾使用另一台电脑直接上传的源码构建，但这些改动没有进入 Git。2026-07-23 部署 `22a372fcd` 后，运行版本与仓库重新一致，同时覆盖了旧源码中的自定义功能和页面。

已确认升级前源码包 `/opt/sub2api-wt3.tar.gz` 可用。以升级前仓库提交 `8a3316441` 为共同基线比较，该源码包包含 74 个自定义文件差异，约 4361 行新增和 1306 行删除；当前分支完整保留 0 个，其中 48 个回退到基线，26 个已被新版改动覆盖。

## 产品分析

### 目标用户与场景
- **用户群体:** 散户站管理员、团队负责人、参与抽奖和兑换活动的终端用户。
- **使用场景:** 管理员投放抽奖次数兑换码、发布仪表盘公告、管理团队与抽奖活动；用户查看团队数据并参与抽奖。
- **核心痛点:** 已上线能力在升级后消失，且旧源码不能直接覆盖当前版本，否则会再次丢失 v0.1.163 的修复。

### 价值主张与成功指标
- **价值主张:** 在保留当前 v0.1.163 能力的前提下恢复已验证的散户站定制功能。
- **成功指标:** 抽奖次数兑换码和仪表盘公告恢复；团队、抽奖及相关后台页面恢复旧版信息结构；当前分支新增能力和测试不回退；前后端构建通过。

### 人文关怀
兑换码发放保持幂等与可审计，避免重复奖励；公告使用纯文本，避免脚本注入；恢复过程不触碰生产数据和运行容器。

## 变更内容
1. 恢复 `lottery_chance` 兑换码类型、奖池字段、生成/兑换流程和审计流水。
2. 恢复可配置的仪表盘醒目公告、用户问候和管理端预览。
3. 恢复团队用户页、团队管理页、抽奖用户页、抽奖管理页及相关仪表盘的旧版页面结构。
4. 将旧 migration `188_redeem_code_pool_key.sql` 重编号为 `191_redeem_code_pool_key.sql`，不修改当前官方 `188-190`。
5. 恢复相关测试、国际化、知识库记录；不恢复仅供旧机器本地开发的端口覆盖文件。

## 影响范围
- **模块:** 兑换码、抽奖、系统设置、用户仪表盘、团队、管理后台。
- **文件:** Go/Ent 后端、SQL migration、Vue/TypeScript 前端、i18n、测试与知识库。
- **API:** 扩展兑换码生成、查询和兑换数据结构；公共设置增加 `dashboard_notice`。
- **数据:** `redeem_codes` 增加可空 `pool_key`，仅允许 `normal` 或 `luxury`。

## 核心场景

### 需求: 抽奖次数兑换码
**模块:** 兑换码 / 抽奖

#### 场景: 管理员生成并由用户兑换
管理员选择普通或豪华奖池并生成指定次数的兑换码。
- 兑换码保存类型、奖池和次数。
- 用户兑换后本人对应奖池的额外次数增加并写入幂等流水。
- 抽奖次数码不触发、也不占用邀请关系的首次兑换奖励。

### 需求: 仪表盘公告
**模块:** 系统设置 / 仪表盘

#### 场景: 管理员配置公告
管理员在通用设置中保存纯文本公告。
- 用户仪表盘顶部展示公告，空内容不展示。
- 管理员仪表盘提供一致的公告预览。

### 需求: 恢复旧版页面
**模块:** 团队 / 抽奖 / 仪表盘

#### 场景: 页面能力恢复且新版行为不回退
- 团队页恢复资格、资金池、统计、治理、成员和消费明细布局。
- 抽奖页恢复奖池主题、次数、转盘、结果、奖品和历史布局。
- 管理页恢复团队审核、详情、资金流水以及抽奖奖池、奖品、规则和流水视图。
- 继续使用当前 API、Pinia 和共享组件，不覆盖新版业务逻辑。

## 风险评估
- **风险:** 旧 migration 编号与官方 `188-190` 冲突。
- **缓解:** 新增 `191`，不修改任何已存在 migration。
- **风险:** 26 个文件同时含旧定制与新版改动，直接复制会回退修复。
- **缓解:** 以 `8a3316441`、旧源码包、当前 `22a372fcd` 做语义三方合并。
- **风险:** 旧页面体积较大，整页替换难以审计。
- **缓解:** 按页面逐个移植，复用当前组件和数据接口，每组完成后立即构建验证。
