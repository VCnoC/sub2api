# Proposal: 团队治理与资金防滥用 (Proposal)

## 1. Context & Problem Statement
- **Current State**: 登录用户可即时创建团队，输入邀请码会立即加入；团队没有人数门槛、创建审核、扩容规则或管理员管理入口。团队资金允许成员存入全部余额，owner 可将资金分配给成员，赠送余额因此可被批量归集。
- **Pain Points**: 管理员无法掌握团队数量、成员规模与资金流向；注册机可利用注册/邀请奖励创建账号并归集团队资金；团队扩容没有与真实充值和实际消费挂钩。

## 2. Value Proposition
- 让管理员统一配置团队创建与升级条件，完整审核创建、超 40 人扩容和异常团队。
- 通过加入审批、人数上限和资金来源额度，阻止赠送余额被自动化账号批量归集。
- 保留现有团队和团队资金功能，避免上线后破坏当前成员关系与余额。

## 3. Alternatives Considered
- **Option A**: 独立治理表、原生 SQL 仓储并兼容现有 Team 服务（采用）。不重新生成 Ent，能避开当前并行开发的 Ent 变更，且便于审计和事务控制。
- **Option B**: 直接扩展 Ent Team/User schema。类型体验更统一，但会与当前未提交的 Ent 生成文件形成大面积冲突。
- **Option C**: 只在前端隐藏创建和转赠入口。无法防止直接调用 API，也不能形成管理员审核和资金来源审计。

## 4. Success Metrics
- [x] 新团队必须经管理员批准，创建门槛和豁免原因可审计
- [x] 邀请码加入改为 owner 审批，批准时严格校验人数上限
- [x] 5/15/40 档条件由管理员配置，owner 主动升级到当前满足的最高档
- [x] 超过 40 人必须申请，管理员可调整目标人数后批准
- [x] 管理员可查看全部团队、成员、指标和资金流水，并可冻结、移除成员、修改人数上限
- [x] 新增余额按来源控制可转赠额度，团队转入资金不可再次转赠
- [x] 现有团队保持成员和人数，复审前不能自动扩容
