# Proposal: 站内工单系统 (Proposal)

## 1. Context & Problem Statement
- **Current State**: 系统已有用户登录、管理员权限、站内公告和 SMTP 通知，但没有用户与管理员持续沟通的站内支持渠道。
- **Pain Points**: 用户只能通过站外方式反馈账号、计费、API 或模型问题；处理过程无法归档、认领、标记优先级，也缺少统一的未读提醒和附件管理。

## 2. Value Proposition
- 为登录用户提供可追踪的双向支持入口，减少问题散落在即时通信工具中的情况。
- 让全部管理员共享工单视图，并通过可选负责人、内部备注、状态和优先级降低重复处理成本。
- 复用现有账号、管理员、邮件和后台任务能力，不引入外部工单平台或新的基础设施。

## 3. Alternatives Considered
- **Option A**: 接入外部工单 SaaS。未采用：需要同步用户身份、工单状态和附件权限，增加外部依赖与数据泄露面。
- **Option B**: 仅提供单次反馈表单。未采用：无法双向追问、认领、关闭或保留完整处理记录。
- **Option C**: 在现有系统内实现最小工单模块。采用：可直接复用登录鉴权、管理员权限、SMTP 和持久化目录，满足当前需求且边界清晰。

## 4. Success Metrics
- [x] 登录用户可创建、查看并持续回复自己的工单
- [x] 管理员可筛选、认领、回复、内部备注、设置优先级和关闭工单
- [x] 新回复可正确切换待处理状态，已关闭工单被用户回复后自动重开
- [x] 站内未读和邮件通知按照负责人规则准确送达
- [x] 附件始终私有访问，并在工单关闭 30 天后自动清理
- [x] 创建限流、权限隔离和文件校验无法被并发请求绕过
